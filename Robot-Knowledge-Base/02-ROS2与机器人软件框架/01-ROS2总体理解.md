# 01-ROS2总体理解

> 学习目标：先把 ROS2 看成“机器人软件系统的通信骨架 + 工程组织方式”，不用一开始陷入源码细节。学完本文后，应能说清 ROS2 解决什么问题、常见模块是什么、机器人系统中的数据如何流动。

---

## 1. 一句话解释

**ROS2（Robot Operating System 2）不是传统意义上的操作系统，而是一套机器人软件开发框架。**

它主要解决：

```text
多个机器人软件模块之间，如何通信、如何组织、如何启动、如何调试、如何复用。
```

可以把 ROS2 理解为机器人系统里的“软件总线”。传感器、定位、规划、控制、执行器等模块都可以作为独立节点运行，通过 ROS2 进行数据交换。

---

## 2. ROS2 在机器人系统中的位置

典型机器人系统链路：

```text
传感器驱动
  ↓
感知 / 状态估计
  ↓
定位 / 建图 / 目标识别
  ↓
任务规划 / 运动规划
  ↓
控制器
  ↓
电机 / 舵机 / 执行器
```

在 ROS2 中，上面每一类功能都可以拆成一个或多个 **Node**：

```text
雷达驱动节点 → SLAM节点 → Nav2规划节点 → 控制节点 → 底盘驱动节点
相机驱动节点 → 目标检测节点 → 抓取规划节点 → 机械臂控制节点
IMU/编码器节点 → 状态估计节点 → 步态控制节点 → 关节驱动节点
```

ROS2 的作用不是替你完成算法，而是提供算法之间的连接方式。

---

## 3. 为什么机器人需要 ROS2

机器人系统通常有这些特点：

1. **模块很多**：相机、雷达、IMU、电机、定位、规划、控制、可视化。
2. **数据类型复杂**：图像、点云、位姿、速度、关节角、地图、轨迹。
3. **实时性要求不同**：图像可能 30Hz，雷达 10Hz，控制可能 100Hz 以上。
4. **需要可视化和调试**：查看 topic、TF、地图、路径、机器人模型。
5. **需要复用开源生态**：Nav2、MoveIt2、ros2_control、SLAM Toolbox、robot_state_publisher 等。

如果不用 ROS2，每个模块都要自己设计通信协议、启动方式、日志、参数、调试工具，工程复杂度会很高。

---

## 4. ROS2 的核心组成

| 概念 | 一句话理解 | 机器人例子 |
|---|---|---|
| Node | 一个独立运行的软件模块 | 雷达驱动节点、导航节点、机械臂控制节点 |
| Topic | 连续数据流通信 | `/scan`、`/odom`、`/cmd_vel`、`/joint_states` |
| Service | 一问一答式通信 | 重置仿真、切换模式、请求保存地图 |
| Action | 长时间任务通信，可反馈、可取消 | 导航到目标点、机械臂执行轨迹 |
| Message | Topic 传输的数据格式 | Twist、LaserScan、Image、JointState |
| Parameter | 节点运行参数 | 速度上限、机器人半径、PID 参数 |
| Launch | 一次性启动多个节点 | 启动机器人模型、仿真、控制器、RViz |
| TF | 坐标系关系管理 | map → odom → base_link → camera_link |
| Package | 功能包，代码组织单位 | nav2、moveit2、turtlebot3_gazebo |
| Workspace | 工作空间，存放多个 package | `~/ros2_ws/src` |

---

## 5. ROS2 的软件架构理解

### 5.1 从节点视角看

```text
Node A 发布数据 → Topic → Node B 接收数据
Node C 发出请求 → Service → Node D 返回结果
Node E 发送目标 → Action → Node F 执行任务并持续反馈
```

### 5.2 从机器人视角看

以 AMR 为例：

```text
/scan        激光雷达数据
/odom        里程计数据
/tf          坐标变换
/map         地图
/goal_pose   目标位姿
/cmd_vel     底盘速度指令
```

数据链路：

```text
雷达 / 编码器 / IMU
    ↓
SLAM / 定位
    ↓
Nav2 全局规划 + 局部规划
    ↓
/cmd_vel
    ↓
底盘驱动
```

以机械臂为例：

```text
/joint_states                 关节当前状态
/robot_description            机器人模型
/planning_scene               规划场景
/trajectory_controller        轨迹控制器
/follow_joint_trajectory      轨迹执行 action
```

数据链路：

```text
目标位姿
  ↓
MoveIt2 逆运动学 / 轨迹规划 / 碰撞检测
  ↓
关节轨迹
  ↓
ros2_control
  ↓
关节驱动器
```

## 6. ROS2 与 DDS 的关系

ROS2 底层通信默认基于 **DDS（Data Distribution Service）**。

你可以先这样理解：

```text
ROS2 提供机器人开发接口
DDS 负责底层数据发现、发布、订阅和传输
```

初学阶段不需要深入 DDS 源码，只需要知道：

- ROS2 节点之间不一定需要中心服务器；
- 节点可以自动发现彼此；
- QoS 会影响通信可靠性、延迟和历史数据保留；
- 多机通信时，网络、防火墙、DDS 配置可能影响节点发现。

---

## 7. ROS2 与 ROS1 的主要差异

| 对比项 | ROS1 | ROS2 |
|---|---|---|
| 通信基础 | ROS Master + TCPROS/UDPROS | DDS |
| 中心节点 | 依赖 roscore | 不需要 roscore |
| 多机通信 | 可用，但配置较麻烦 | 原生考虑分布式 |
| 实时性 | 支持有限 | 更重视实时性和工业应用 |
| 生命周期管理 | 较弱 | 支持 Lifecycle Node |
| 安全性 | 较弱 | 支持 DDS-Security 方向 |
| 当前学习价值 | 仍有大量老项目 | 新项目建议优先 ROS2 |

你的学习目标是机器人新项目和仿真，因此建议以 ROS2 为主。

## 8. ROS2 常见目录结构

一个 ROS2 工作空间通常长这样：

```text
ros2_ws/
├─ src/
│  ├─ my_robot_bringup/
│  ├─ my_robot_description/
│  ├─ my_robot_control/
│  └─ my_robot_navigation/
├─ build/
├─ install/
└─ log/
```

说明：

| 目录 | 作用 |
|---|---|
| `src/` | 存放源码和功能包 |
| `build/` | 编译过程产生的中间文件 |
| `install/` | 编译安装后的可运行文件 |
| `log/` | 编译和运行日志 |

常用命令：

```bash
mkdir -p ~/ros2_ws/src
cd ~/ros2_ws
colcon build
source install/setup.bash
```

## 9. ROS2 常用命令速查

### 9.1 环境检查

```bash
printenv | grep ROS
ros2 --help
ros2 doctor
```

### 9.2 功能包

```bash
ros2 pkg list
ros2 pkg executables
ros2 pkg prefix <package_name>
```

### 9.3 节点

```bash
ros2 node list
ros2 node info /node_name
```

### 9.4 Topic

```bash
ros2 topic list
ros2 topic list -t
ros2 topic echo /topic_name
ros2 topic info /topic_name
ros2 topic hz /topic_name
ros2 topic pub /topic_name <msg_type> "data"
```

### 9.5 Service

```bash
ros2 service list
ros2 service type /service_name
ros2 service call /service_name <srv_type> "request"
```

### 9.6 Action

```bash
ros2 action list
ros2 action info /action_name
ros2 action send_goal /action_name <action_type> "goal"
```

### 9.7 Launch

```bash
ros2 launch <package_name> <launch_file.py>
```

## 10. 第一个建议实验：turtlesim

### 10.1 实验目标

通过 turtlesim 理解：

```text
Node → Topic → Service → 参数 → 调试工具
```

### 10.2 安装

以 ROS2 Humble 为例：

```bash
sudo apt update
sudo apt install ros-humble-turtlesim
```

如果你使用的是 Jazzy、Iron 或其他版本，把 `humble` 替换成你的 ROS2 版本名。

### 10.3 运行

终端 1：

```bash
source /opt/ros/humble/setup.bash
ros2 run turtlesim turtlesim_node
```

终端 2：

```bash
source /opt/ros/humble/setup.bash
ros2 run turtlesim turtle_teleop_key
```

### 10.4 观察节点

```bash
ros2 node list
```

你会看到类似：

```text
/turtlesim
/teleop_turtle
```

### 10.5 观察 Topic

```bash
ros2 topic list
ros2 topic echo /turtle1/cmd_vel
```

按键控制乌龟移动时，`/turtle1/cmd_vel` 会出现速度指令。

### 10.6 观察 Service

```bash
ros2 service list
ros2 service call /clear std_srvs/srv/Empty "{}"
```

这会清除乌龟轨迹。

## 11. ROS2 学习时最重要的思维方式

### 11.1 不要只记命令，要看数据链路

每看到一个 ROS2 项目，先问：

```text
有哪些节点？
每个节点输入什么？
每个节点输出什么？
关键 topic 是什么？
有没有 service/action？
用什么 launch 启动？
参数文件在哪里？
```

### 11.2 不要一开始读源码

建议顺序：

```text
README → launch 文件 → config 参数 → topic/action/service → 节点入口 → 核心算法
```

### 11.3 对每个项目画一张图

示例：

```text
camera_node → /image_raw → detector_node → /target_pose → planner_node → /cmd_vel
```

## 13. 常见误区

| 误区 | 正确理解 |
|---|---|
| ROS2 是操作系统 | ROS2 是机器人软件框架，不是 Ubuntu/Windows 这种 OS |
| 学 ROS2 就是学算法 | ROS2 更多是工程组织和通信框架，算法通常在节点内部 |
| 先读源码才能学会 | 初学先跑通 demo、看 topic、画链路图更重要 |
| Topic、Service、Action 差不多 | 三者适用场景不同，必须区分 |
| Launch 只是启动脚本 | Launch 是机器人系统工程化启动的核心入口 |
| TF 可有可无 | 多传感器、多关节机器人里 TF 非常关键 |

---

## 14. 本文总结

ROS2 的核心价值：

```text
把复杂机器人系统拆成多个节点，
再通过 topic、service、action、parameter、TF、launch 等机制连接起来。
```

你当前阶段只需要掌握：

1. ROS2 解决什么问题；
2. Node / Topic / Service / Action 的区别；
3. 如何用命令观察系统运行状态；
4. 如何通过 launch 启动多个节点；
5. 如何把 ROS2 放进 AMR、机械臂、四足、双足等机器人系统链路里理解。
