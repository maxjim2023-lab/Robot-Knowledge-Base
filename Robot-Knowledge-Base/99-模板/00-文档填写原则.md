# 文档填写原则

本文档用于约束 Robot-Knowledge-Base 中各类笔记的填写方式。后续补全文档前，应先阅读本文件，再根据具体主题选择对应模板。

## 1. 总目标

知识库的目标不是复制资料，也不是写成教材，而是形成“6 成了解”级别的机器人系统认知：

- 能说清概念是什么、解决什么问题、输入输出是什么；
- 能理解该概念在机器人系统链路中的位置；
- 能把感知、规划、控制、执行器之间的关系讲清楚；
- 能记录实际实验、报错、参数变化和个人理解；
- 能持续扩展，而不是一次性写满。

## 2. 填写深度

每篇文档默认控制在“能讲清主干，不深挖所有细节”的程度。

优先写清：

- 一句话解释；
- 它解决什么问题；
- 输入是什么；
- 处理过程是什么；
- 输出是什么；
- 在机器人系统中的位置；
- 典型应用；
- 容易混淆的概念；
- 当前还不理解的问题。

暂时不要追求：

- 从零推导复杂公式；
- 逐行分析源码；
- 罗列大量未经消化的资料；
- 写成百科词条；
- 过早深入高级算法实现细节。

## 3. 推荐文档结构

通用概念类文档优先使用以下结构：

```markdown
# 标题

## 1. 一句话解释

## 2. 它解决什么问题

## 3. 输入-处理-输出

## 4. 在机器人系统中的位置

## 5. 关键概念

## 6. 典型应用

## 7. 容易混淆的概念

## 8. 我的理解

## 9. 待深入问题
```

如果是实验、项目阅读、命令速查、硬件认知等特殊文档，应优先使用 `99-模板` 中对应模板。

## 4. 自动填写与手动填写分工

自动填写适合完成：

- 概念解释；
- 基础背景；
- 系统链路；
- 输入-处理-输出；
- 常见模块关系；
- 初版 Mermaid 图；
- 常见问题清单；
- 开源项目阅读框架。

手动填写更适合完成：

- 实际运行环境；
- 真实命令输出；
- 截图、视频和现象；
- 具体报错；
- 参数修改后的变化；
- 自己真正想通的理解；
- 学习当天的困惑。

自动填写的内容应被视为“可修改初稿”，不是最终结论。

## 5. 每篇文档的写作原则

- 先写主干，再补细节；
- 先写自己的理解，再补资料链接；
- 每个概念尽量按“输入-处理-输出”拆解；
- 每个机器人方向都要能落到“感知 -> 规划 -> 控制 -> 执行”链路；
- 重要概念要建立 Obsidian 双链；
- 不确定内容要明确标注“待确认”或放入问题库；
- 学到新内容时优先更新已有文档，而不是重复新建相似文档；
- 实验过程中出现的问题要同步沉淀到 `14-问题库`。

## 6. 各类文件填写重点

### 总览与路线图

重点是建立导航、索引和全局理解，不写过多细节。

应包含：

- 学习目标；
- 阶段路线；
- 核心概念索引；
- 开源项目索引；
- 待深入问题。

### 机器人基础

重点是解释通用底层概念，例如坐标系、自由度、运动学、动力学、传感器、执行器和控制基础。

应避免写得过深，优先服务后续 AMR、机械臂、四足、双足等专题。

### ROS2 与机器人软件框架

重点是理解机器人软件系统如何组织。

应突出：

- Node、Topic、Service、Action 的区别；
- TF 在坐标变换中的作用；
- Launch 如何组织启动；
- RViz 如何观察系统状态；
- ros2_control 如何连接软件控制器和硬件接口；
- 常用命令和常见报错。

### 仿真软件

重点是比较 Gazebo、MuJoCo、Isaac Sim / Isaac Lab 的适用场景。

应记录：

- 适合哪类机器人；
- 物理仿真特点；
- 与 ROS2 的关系；
- 安装和运行问题；
- 实验现象。

### AMR、机械臂、四足、双足与人形机器人

每个方向都应至少形成三类内容：

- 总体架构；
- 核心概念；
- 实验或项目笔记。

写作时优先回答：

- 这个机器人由哪些模块组成；
- 传感器输入是什么；
- 规划或控制模块做什么；
- 最终输出到什么执行器；
- 该方向最大的工程难点是什么。

### 感知与智能控制

重点是理解视觉、听觉、触觉如何变成机器人可用的目标、状态或语义信息。

不要把智能控制写成“直接控制电机”，应强调它通常输出目标、策略、动作意图或高层任务。

### 控制与强化学习

重点是理解 PID/PD、阻抗控制、MPC、WBC、RL、Sim2Real 的基本含义和适用位置。

公式可以少量出现，但必须服务于直觉理解。

### 问题库

问题库用于沉淀真实学习过程中的阻塞点。

每个问题应尽量记录：

- 问题描述；
- 发生场景；
- 已尝试方法；
- 当前状态；
- 最终解决方案；
- 复盘经验。

## 7. 链接与索引原则

- 重要概念第一次出现时尽量使用 Obsidian 双链；
- 跨主题相关内容要互相链接；
- 总览文件只做导航，不堆正文；
- 项目笔记要链接到相关概念笔记；
- 实验记录要链接到相关问题记录；
- 周复盘要链接到本周新增或更新的知识文档。

## 8. Mermaid 图原则

能画链路图时优先用 Mermaid。

常用链路：

```text
感知 -> 规划 -> 控制 -> 执行
```

机器人方向示例：

```text
AMR：雷达/相机/IMU/编码器 -> SLAM/定位 -> 全局规划 -> 局部避障 -> cmd_vel -> 底盘电机
机械臂：目标位姿 -> IK -> 轨迹规划 -> 碰撞检测 -> 关节轨迹 -> 关节控制器
四足：IMU/编码器/足端接触 -> 状态估计 -> 步态规划 -> 足端轨迹 -> IK/WBC -> 关节控制
双足：IMU/编码器/足底接触 -> 质心估计 -> 支撑状态判断 -> 摆动腿规划 -> WBC -> 关节输出
```

## 9. 质量检查清单

填写或更新文档后，应快速检查：

- 标题是否清晰；
- 是否有一句话解释；
- 是否写了输入-处理-输出；
- 是否说明了系统位置；
- 是否有“我的理解”或“待深入问题”；
- 是否和相关文档建立链接；
- 是否避免了大段复制资料；
- 是否把实验问题同步到问题库。

## 10. 后续填写默认流程

后续补全文档时默认按以下流程执行：

```text
读取文档填写原则 -> 读取相关模板 -> 查看目标文件现状 -> 补骨架 -> 填初版内容 -> 建立链接 -> 更新问题/索引
```

如果用户明确要求只创建骨架，则只写结构，不自动补充正文内容。

如果用户要求自动填充，则先按“6 成了解”写初稿，再保留“我的理解”和“待深入问题”供后续继续完善。
